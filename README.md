## 核心防护机制（为什么难杀）
学生机房管理助手采用了多层互相守护 + 随机重命名策略：

1. **主要进程**：
   - `jfglzs.exe`：主程序 / 守护进程（启动后常被重命名）
   - `przs.exe`：核心保护进程（启动后会被重命名为随机名称）
   - `zmserv.exe`：最核心的系统服务进程（伪装成关键系统服务）
     - 它会**周期性检测**其他几个进程是否存在
     - 如果检测到缺失 → **立即关机** 或 **蓝屏锁定**
     - 难以直接结束（权限高、自我保护强）

2. **随机重命名机制**（从 v9.x+ 开始加强）：
   - 安装路径：`C:\Program Files (x86)\`
   - 每天/每次重启会在此目录下**新建一个随机文件夹**（文件夹名：2~6个字符，字母+数字混合）
   - 在该随机文件夹内生成一个**随机命名的 .exe**（文件名：2~6个字符）
   - 这个随机 exe **实际上就是 przs.exe 的副本**（内容完全相同）
   - 随机 exe 会反过来守护 `jfglzs.exe`
   - 示例路径：`C:\Program Files (x86)\bcd\abcde.exe`（实际名称随机）

3. **守护关系**（循环保护）：
   - jfglzs.exe 守护 zmserv.exe + przs.exe（随机进程）
   - 随机进程 守护 jfglzs.exe
   - zmserv.exe 守护全部 → 缺失任何一个就强制关机

## 清理原理（本工具采用的方法）

核心思路：**让 zmserv.exe “以为” przs.exe 等进程还在，但实际上已经失效**。

具体步骤：
1. **枚举并杀死非核心进程**  
   快速结束 `jfglzs.exe` 和 **随机命名的 przs.exe 副本**（不碰 zmserv.exe）。

2. **立即在原随机位置替换占位文件**  
   - 找到当前正在运行的随机 przs.exe 的**完整路径**
   - 用一个**空白但有效的、可运行的 exe 文件**覆盖它（大小很小，内容可以是简单的 hello world 程序）
   - 占位文件要求：
     - 必须是**有效的 PE 可执行文件**（不能是 0 字节空文件，否则 zmserv 可能检测异常）
     - 文件名和路径保持不变（zmserv 只检查路径+文件名是否存在，不深度校验内容）
     - 建议用一个**永不退出**的小程序（死循环或 Sleep）占位，防止被其他机制清理

3. **结果**：
   - zmserv.exe 检测到“进程文件还在” → 不触发关机/蓝屏
   - 但实际守护逻辑已经失效 → 小助手功能被彻底删除

## 本工具使用方法
一、**删除**
1. 以**管理员权限**运行本工具（必须！）
2. 工具会自动：
   - 扫描 `C:\Program Files (x86)\` 下所有 2~6 字符长度的子文件夹
   - 在这些文件夹中查找 2~6 字符的 .exe 文件
   - 判断哪些是正在运行的进程（匹配 przs.exe 的特征或直接查进程列表）
   - 结束 jfglzs.exe + 所有匹配的随机进程
   - 快速用占位 exe 覆盖原随机 exe 文件
3. 成功后会看到提示“删除”
4. 可选：重启电脑验证（大部分情况下无需重启即可看到效果）

二、**还原**

   为了方便测试/回滚，工具在破解时会自动备份原随机 exe 文件为：
   - `原路径\原文件名.exe.bak`（例如 `C:\Program Files (x86)\x7k9p\q2m4.exe.bak`）

   还原操作：
   - 运行工具 → 选择“还原”模式
   - 自动扫描所有 `*.exe.bak` 文件
   - 将 `.bak` 文件重命名为原文件名（覆盖占位文件）
   - 恢复守护链 → 小助手重新生效（需大约500毫秒生效）  

这样即使多次破解/测试，也能随时恢复原状。
## 注意事项 & 风险
- **还原卡 / 深信服 / 冰点**：如果机房有硬盘还原，重启后会恢复，需在还原前操作或配合其他工具
- **新版本可能加强检测**：如 v12.97+ 可能对占位文件做哈希或大小校验（目前钟鸣还没做此类更改）
- **法律风险**：在学校机房使用可能违反校规，甚至更严重的规定，请自行判断
- **不保证 100% 成功**：不同学校部署版本、系统补丁、其他防护软件会影响效果
